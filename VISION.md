# CollabAI 愿景

## 我们为什么做这个

现在的 AI 编程工具，本质上在模仿"IDE + 智能提示"的范式。它们的底层假设是：开发者是独立的、自足的、只需要更快的打字速度。

这个范式在单人或小团队（2-3 人）时确实高效，但一旦团队规模超过阈值，**沟通成本就开始指数级吞噬开发效率**——而这恰恰是大型软件项目的常态。

> **代码是沟通的最终产物，但现在的 AI 工具只优化了"写代码"这个末端动作，却完全忽略了"达成共识"这个前置过程。**

## 我们要做什么

一个 **AI 技术协作者**——不只是回答问题，而是项目的"技术负责人"角色：

### 1. 风格翻译器
A 写了一段代码，AI 知道 B 的模块需要什么接口，自动建议适配层。

### 2. 知识同步器
A 发现某个第三方库有坑，AI 记录后，下次 B 想用时主动提醒。

### 3. 进度协调者
"B 你依赖的 A 的接口还要 2 天，要不你先 mock 着？"

### 4. 代码考古学家
"这个奇怪的设计是 3 个月前 A 为了兼容某个旧版本加的，现在可以删了。"

### 5. 冲突调解者
"A 你改了 create_order 的返回值，B 正在写调用这个函数的代码。建议先同步。"

## AI 技术协作者的一天

**早上 9:00** — AI 推送："昨晚 user-service 有 3 个 PR 合并，涉及认证流程变更。order-service 开发者请注意：validate_token() 返回值从 bool 改成了 TokenPayload。"

**上午 10:30** — 开发者 A 问："我要加一个 webhook，放哪合适？" AI："根据项目规范，统一放在 integrations/webhooks/。另外 B 上周也加了 webhook，用的是 fastapi-webhooks 库，建议保持一致。"

**中午 12:00** — AI 检测到 A 和 C 同时修改了 models/order.py："A 加了 cancellation_reason，C 加了 refund_status。两者都涉及订单状态机，建议统一放在 OrderStatusHistory 表里。"

**下午 3:00** — 新人 D 提交代码，AI 在 PR 评论："逻辑正确，但风格不一致。项目约定使用早期返回减少嵌套，异常处理用自定义异常。"

**下午 5:00** — AI 每日总结："今日变更：4 个 PR，2 个跨服务调用。新增技术债：A 临时绕过缓存层（已标记 TODO #142）。"

## 技术可行性

不需要等 AGI。现在就可以做到：

| 能力 | 现有技术 |
|------|---------|
| 理解整个代码库 | Tree-sitter + 向量索引 |
| 跟踪谁在改什么 | Git hooks + 文件系统监控 |
| 理解开发者风格 | 静态分析历史提交 |
| 检测冲突意图 | AST 差异分析 + LLM 语义理解 |
| 主动协调建议 | 实时消息推送（Slack bot） |
| 维护项目认知 | 结构化记忆库 |

**真正的瓶颈不是技术可行性，而是产品形态。**
